# Overview
-  USART2 is configured to provide a serial link between the Nucleo-F401RE and host PC via the on-board ST-LINK virtual COM port. The primary use case for UART connection is to transmit calibrated sensor readings and debug prints to a terminal emulator during development. 
# Hardware Configuration 

|  Signal   | Pin | AF  |          Notes          |
| :-------: | :-: | :-: | :---------------------: |
| USART2_TX | PA2 | AF7 | Push-pull, medium speed |
| USART2_RX | PA3 | AF7 |   Input with pull-up    |

|  Parameter   |               Value                |
| :----------: | :--------------------------------: |
|  Peripheral  |         USART2 (APB1 Bus)          |
| Clock source |    PCLK1 (HSI default - 16MHz)     |
|  Baud rate   |               115200               |
| Frame format | 8 data bits, no parity, 1 stop bit |
| Oversampling |                x16                 |
| Flow control |                None                |
-  The ST-LINK provides a virtual COM port that is routed directly to USART2 peripheral on the target MCU. No external USB-UART adapter is required.
# Design Decisions 
-  Two design approached are viable for this application. A polling or interrupt driven approach each with strengths and costs. 
### Polling (blocking Tx)
-  The CPU actively waits for the `TXE` flag to be set by hardware, then writes the next byte to the data register. This repeats until all the bytes are sent. The `TC` flag can also be polled after the final byte to confirm that the last stop bit has physically left the shift register. 
	-  Strengths : Simple to implement
	-  Costs : CPU is stalled for every byte. Unsuitable for large or continuous transfers; will block main loop. 
### Interrupt-driven Tx
-  The producer function copies data into a ring buffer and return immediately. `TXEIE` is enabled only when the buffer transitions from empty to non-empty. When both `TXE` and `TXEIE` are asserted, the hardware fires and interrupt and the ISR drains the buffer one byte per interrupt. `TXEIE` is disabled when the buffer become empty, preventing continuous interrupts since `TXE` remains high while the data register is idle, as DR is empty.
	-  Strength : CPU involvement is minimal - ISR only runs when the hardware is ready for the next byte. 
	-  Cost : Increased complexity; requires ring buffer and has a few critical sections. 
-  **Decision** : The interrupt-driven approach is used as the production implementation. Polling implementation was also built as a learning exercise but will not be explained further nor retained in source files.  
# State Machine 
-  Interrupt-driven Tx path follows a three-state machine : 
![](../../Images/usart2_state_machine.png)

| State         | Conditions                                      |
| ------------- | ----------------------------------------------- |
| `IDLE`        | `tx_active = 0`, `TXEIE = 0`, buffer empty      |
| `TX_ACTIVE`   | `tx_active = 1`, `TXEIE = 1`, buffer has data   |
| `ISR_RUNNING` | CPU inside `USART2_IRQHandler`, draining buffer |
-  Transitions between states : 
	-  `IDLE -> TX_ACTIVE` : `uart2_write_bytes()` called, data is enqueued in the ring buffer, `tx_active` is set and `TXEIE` is enabled. Since `TXE` will already be high (DR empty), the interrupt fires immediately once the NVIC is re-enabled.
	-  `TX_ACTIVE -> ISR_RUNNING` : Hardware asserts the interrupt line when `TXEIE` and `TXE` are both high. The cortex-M4 performs an exception entry, saves context to the stack, and vectors to `USART2_IRQHandler`.
	-  `ISR_RUNNING -> TX_ACTIVE` : If the buffer is not empty (`head!= tail`), `TXEIE` is cleared and `tx_active` is set low. This prevents an interrupt storm since `TXE` remain asserted while DR is empty.
	-  `TX_ACTIVE` self loop : If `uart2_write_bytes()` is called while a transmission is already in progress (`tx_active = 1`), the function enqueues new data only - `TXEIE` is already set so no additional kicks are needed.

# Initialisation 
-  USART2 initialisation is performed in four stages. 
### Stage 1 -  Enable Peripheral Clocks
-  The GPIOA and USART2 clock gates are enabled before any register access. 
	-  Enable GPIOA clock : `RCC->AHB1ENR`
	-  Enable USART2 clock : `RCC->APB1ENR`
-  Dummy reads are performed after each enable write. This ensure the clock enable has propagated through the bus before the first peripheral register access. This stage is recommended in the reference manual. 
### Stage 2 - Configure the GPIO Pins
-  PA2 (Tx) : 
	-  Mode - Alternate function : `GPIOA->MODER`
	-  Output type - push-pull :  `GPIOA->OTYPER`
	-  Speed - medium : `GPIOA->SPEEDR`
	-  Pull - None : `GPIOA->PUPDR`
	-  Alternate function - AF7 / USART2 : `GPIOA->AFR[0]` or `GPIOA->AFRL` (called AFRL in RM but use first in code).
-  PA3 (Rx) : 
	-  Mode - AF : `GPIOA->MODER`
	-  Pull - Pull-up : `GPIOA->PUPDR`
	-  Alternate function - AF7 / USART2 : `GPIOA->AFR[0]`
### Stage 3 - Configure USART2 Peripheral 
-  The peripheral is disabled, `UE = 0`, before configuration to allow register writes. 
	-  Oversampling - x16 : `USART2->CR1`
	-  Word length - 8 data bits : `USART2->CR2`
	-  Parity - None : `USART2->CR1`
	-  Stop bits - 1 : `USART2->CR2`
-  Baud rate calculation occurs and written to BRR. An incorrect value will produce garbled output. Using x16 oversample : 
``` C
uint32_t usartdiv_times16 = (pclk + (baud / 2U)) / baud;
uint32_t mantissa = usartdiv_times16 / 16U; // BRR[15:4]
uint32_t fraction = usartdiv_times16 % 16U; // BRR[3:0]

USART2->BRR = (mantissa << 4U) | (fraction & 0xFU); 
```
-  Important note : if using x8 oversample a different calculation is needed, will not be included in driver. 
-  After BRR is written : 
	-  Enable transmitter - `TE = 1` : `USART2->CR1`
	-  Enable USART2 - `UE = 1`  : `USART2->CR1`
### Step 4 - Interrupt Configuration (Tx only)
-  `TXEIE` is left disabled at initialisation and must only be set when data is enqueued. If left enabled with an empty buffer, the peripheral will continuously interrupt since `TXE` remains high when the DR is idle. 
-  Initialise the ring buffer, `head`, `tail` and `tx_active` as static file scope variables. 
-  Configure the NVIC using CMSIS functions : 
	-  Set interrupt priority - low priority 
	-  Clear any pending ISQ
	-  Enable the interrupt line
-  The NVIC is part of the Cortex-M4 core (ARM architecture), not the USART peripheral itself.  
# Interrupt-driven Tx Implementation
-  Two functions form the transmit path : 
```C
uart_status_t uart2_write_bytes(const uint8_t *buf, size_t n) // producer
void USART2_IRQHandler(void) // consumer 
```
-  Shared state (all static file scope) : 
```C
static uint8_t tx_buf[TX_BUF_SIZE];
static uint16_t tx_head;
static uint16_t tx_tail;
static uint8_t tx_active;
```
### Ring Buffer Invariants : 
-  `head == tail` means that the buffer is empty.
-  `head != tail` means that the buffer contains unread data.
-  `head` is the next write position (producer function advances)
-  `tail` is the next read position (consumer function advances)
-  Both wrap around at `TX_BUF_SIZE`.
### Producer Function - `uart2_write_bytes()`
1.  Validate input 
2.  Disable `USART2_IRQn` - creates a critical section to prevent the ISR from modifying `tail` or `tx_active` mid-update. 
3.  Compute free space in the buffer. If `n > free`, re-enable the IRQ and return error state that buffer is full (`UART_STATUS_BUFFER_FULL`).  Else, meaning is enough space in buffer for n bytes, copy these n bytes into `tx_buf` at `head`, advancing and wrapping `head` for each byte.
4.  If `tx_active == 0` (Idle to active transmission)
	-   Set `tx_active = 1`
	-  Enable `TXEIE` : `USART2->CR1`
	-  Since `TXE` is already high (DR is empty), the interrupt will fire as soon as `USART2_IRQn` is re-enabled. 
5.  Re-enable `USART2_IRQn`. 
6.  Return status (`UART_STATUS_OK`)
-  The function returns once all the data is enqueued - not once transmission is complete. This is the key advantage over a blocking solution. 
### Consumer - `USART2_IRQHandler()`
-  This handler is called automatically by hardware via the vector table. For the CPU to enter it, two conditions must be true : 
	-  The NVIC interrupt line must be enabled.
	-  The USART2 peripheral asserts its interrupt line. (flagging condition)
-  On exception entry the cortex-m4 automatically pushes context to the stack, then fetches the handler address from the vector table (vector table typically defined in the startup file). 
-  Handler logic : 
	1.  Check both `TXE` and `TXEIE` are set before handling (Ensure flagging conditions true). This guards against spurious or unrelated USART2 interrupts.
	2.  If `head != tail` means buffer has data : 
		-  Write `tx_buf[tail]` to Data register : `USART2->DR`
		-  Advance and wrap `tail`
		-  Hardware clears `TXE` when DR is written; the next interrupt fires when the next byte has shifted into the shift register and DR is empty again. 
	3.  If `head == tail` means buffer is empty : 
		-  Clear `TXEIE` : `USART2->CR1`
		-  Set `tx_active = 0`
		-  Without this, `TXEIE` remaining high with `TXE` would cause continuous interrupts to fire with an empty buffer (nothing to transmit). 
# Testing and Verification 
-  Testing is simple as printing a `"boot ok"` message from main immediately after `uart2_init()` returns. This validates the full transmit path on every reset. Rather than use a console in cubeIDE create a host setup using picocom. 
```bash 
picocom -b 115200 /dev/ttyACM0
```
-  Can create an `alias - bme280-uart` to simplify this terminal command. Must ensure that the baud rate matches that written to BRR else output will be random.  
-  To exit the console need to use ctrl A followed by ctrl X. 
# Design Notes 
-  **Static storage class** - Ring buffer variables and `tx_active` are declared static at file scope for two reasons :
	-  Persistence - stack allocated locals are destroyed when a function returns. These variables must survive across function calls and ISR invocations for the lifetime of the driver. 
	-  Encapsulation - static at file scope restricts visibility to `uart.c`, preventing other modules from accidentally accessing or corrupting driver internals.
-  **Volatile Qualifier** - Variables shared between the main execution context and the ISR must be declared volatile (eg `head tail tx_active`). Without it the compiler may cache a variables value in a register across an ISR boundary, producing a situation where the main context reads a stale cached value rather than the updated memory location by the ISR or vice versa. `volatile` forces a memory access on every read and write, ensuring both contexts observe each others updates.
