# Overview 
-  I2C1 is configured to provide a synchronous serial link between the nucleo-F401RE and BME280 atmospheric sensor. The primary use case is reading calibrated temperature, pressure and humidity data from the sensor over a two wire bus. Secondary use case is writing to configuration registers on the BME280 at startup to set its operating mode and oversampling settings. 
-  I2C1 is operated in standard mode at 100kHz. An interrupt-driven approach is used so that the CPU is not stalled during multi-byte transfers. The transfer is driven by hardware event flags which are surfaced to the CPU via the I2C1 event and error interrupts.  
#  Hardware Configuration 

|  Signal  | Pin | AF  |              Notes              |
| :------: | :-: | :-: | :-----------------------------: |
| I2C1_SCL | PB8 | AF4 | Open-drain, pull-up, high speed |
| I2C1_SDA | PB9 | AF4 | Open-drain, pull-up, high speed |

|  Parameter   |                              Value                              |
| :----------: | :-------------------------------------------------------------: |
|  Peripheral  |                         I2C1 (APB1 Bus)                         |
| Clock source |                      PCLK1 (HSI - default)                      |
|  Bus speed   |                     100kHz (standard mode)                      |
|  Addressing  |                              7-bit                              |
|   Pull-ups   | External 4kohm (breakout board)  + internal (GPIO) in parrallel |
-  PB8 and PB9 are the dedicated I2C1 pins on the STM32F401RE. Both must be configured as open-drain - I2C standard requires that any device on the bus can only pull lines low; the pull-up resistors return the lines high. 
-  BME280 breakout board provides external 4kOhm pull-up resistors. The internal GPIO pull-ups are also configured for safety but contribute negligibly compared to the external pull-ups. 
# Design Decisions 
-  Only one viable approach for I2C transfers as a polling solution would stall the CPU for the entire transfer - unacceptable. So must use an interrupt driven approach to minimise CPU stall time. 
# State Machine

| State       | Path             | Entry Condition                                |
| ----------- | ---------------- | ---------------------------------------------- |
| IDLE        | Common           | No active transfer, bus released               |
| START_SENT  | Common           | Producer asserted `START`                      |
| ADDR_W_SENT | Common           | `SB` set, SLA+W written to DR                  |
| REG_SENT    | Common           | ADDR cleared, `reg` written to DR              |
| TX_VAL      | Write only       | `TXE` set, `tx_val` written to DR              |
| START_R     | Read only        | `TXE` and `BTF` set, repeated `START` asserted |
| ADDR_R_SENT | Read only        | `SB` set, SLA+R written to DR                  |
| RX_1        | Read rx_len == 1 | `ACK = 0`, `STOP` queued, `ADDR` cleared       |
| RX_2        | Read rx_len == 2 | `POS = 1`, `ACK = 0`, `ADDR` cleared           |
| RX_BULK     | Read rx_len >= 3 | `ACK = 1`, `ADDR` cleared                      |
-  Transitions and flag sequences are documented in detail in the implementation section. 
# Initialisation 
-  I2C1 initialisation is performed in four stages
### Stage 1 - Enable Peripheral Clocks 
-  The GPIOB and I2C1 clock gates must be enabled before any register access to those peripherals can occur. 
	-  Enable GPIOB clock : `RCC->AHB1ENR`
	-  Enable I2C1 clock : `RCC->APB1ENR`
-  Dummy reads required after each write to ensure the clock gate has propagated through the bus before the first peripheral register access (recommended in RM). 
-  After I2C1 clock is enabled and dummy read, a software reset is applied and then released. This guarantees the peripherals start from a known state regardless of previous activity. 
	-  i2c1 software reset : `RCC->APB1RSTR` 
### Stage 2 - Configure GPIO Pins 
-  Both PB8 (SDL) and PB9 (SDA) require identical configuration : 
	-  Mode - Alternate function : `GPIOB->MODER`
	-  Alternate function - AF4/i2c1 : `GPIOB->AFR[1]`
	-  Output type - Open drain : `GPIOB->PUPDR`
	-  Pull - Pull-up : `GPIOB->PUPDR`
	-  Speed - High : `GPIOB->OSPEEDR`
-  Open drain is required, as the i2c protocol communicates by devices pulling the bus low - pull-up resistors passively return the line high. Configuring push-pull can damage the bus and will cause communication errors. Open-drain is mandatory for i2c protocol. 
### Stage 3 - Configure I2C1 Peripheral 
-  The peripheral is disabled before configuration `PE = 0`. 
-   Peripheral clock frequency is set to the APB1 peripheral clock frequency in MHz this value is used internally by the peripheral to generate timing. MUST match APB1 :  `I2C1->CR2`
-  i2c controller mode selection - standard mode : `I2C1->CCR`
-  Clock control register, controls SCL clock period in standard mode with 50% duty cycle formula is : 
```C
uint32_t ccr = (APB1_CLK_HZ / (2U * I2C1_SPEED_HZ));
```
-  where `I2C1_SPEED_HZ` is a macro for 100kHz (the decided bus speed). RM specifies ccr must be greater than 4, for this specification value of 80 is calculated however may want to shield against writing less than 4. RM provides example calculations. 
-  TRISE specifies the maximum allowable rise time of SCL/SDA in peripheral clock tick. Formula as stated in the RM : 
```C
I2C1->TRISE |= ((APB1_CLK_MHZ + 1U) & I2C_TRISE_TRISE); // 17
```
-  After timing registers are written, the peripheral can be re-enabled `PE = 1` and ACK is enabled. ACK must be enabled for the master to acknowledge received bytes during  a read sequence. 
### Stage 4 -  Interrupt and Event Configuration 
-  Two interrupt lines are enabled : the event interrupt and the error interrupt for i2c1 peripheral. Must configure the NVIC using CMSIS functions : 
	-  Set priority - ER higher than EV and both higher than USART2. 
	-  Clear any pending interrupts before enabling.
	-  Enable the interrupt line. 
-  Two bits in `I2C1->CR1` which peripheral conditions raise event interrupt : 
	-  `ITEVTEN` - enables interrupts for sequencing events (eg : SB, ADDR)
	-  `ITBUFEN` - enables interrupts for buffer events (eg : TXE, RXNE)
	-  `ITERREN` - enables interrupts for all error flags (eg : BERR, AF)
-  Unlike `TXEIE` in the USART driver, these bits can be permanently enabled. The I2C peripheral only asserts it interrupt line when a specific bus event has occurred, so there is no risk of a continuous interrupt storm from a persistently set flag.  
# Implementation 
-  Four functions form the transfer path : 
	-  Producer functions - write_reg() & read_regs(). 
	-  Consumer functions - EV_IRQHandler() & ER_IRQHandler() (Event and error driven). 
-  A shared state held in a single static file scope context struct, must also be volatile. Named `handle`. 
### Producer Functions - write_reg() / read_regs() 
-  Software actions : 
	-  Validate inputs and check bus is not busy.
	-  Populate relevant fields in context struct.
	-  Set state to START_SENT
	-  Assert start condition  (`ACK` and `START` for read or `START` for write).
	-  Producer function returns (non-blocking)
-  Hardware actions : 
	-  Detects bus is idle, SDA and SCL both high. 
	-  Generates start condition, pulls SDA low while SCL high. 
	-  Sets `SB` = 1 (START bit flag).
	-  Since `ITEVTEN` is set, this asserts `I2C1_EV_IRQn`, the NVIC fires the event ISR. 
### Consumer Function - Event Handler 
### STATE : START_SENT
-  Software actions : 
	-  ISR entry SR1 is read. 
	-  Check `SB` is high else break. 
	-  Write 7 bit device address with write bit. (`(addr7 << 1) | 0`) to DR. Required as must tell the device which register to read.  
	-  Hardware clears `SB` automatically when DR is written - `SB` cannot be cleared any other way. 
	-  Set state to ADDR_W_SENT
-  Hardware actions : 
	-  Shifts the address byte out onto SDA, 8 bits, MSB first. 
	-  Releases SDA on the 9th clock, the slave (BME280) must pull SDA low to ACK. 
	-  If ACKed : sets ADDR flag to 1, asserts event IRQ. 
	-  If NACKed : sets AF flag to 1, asserts the error IRQ - error handler takes over. 
### STATE : ADDR_W_SENT
-  Software actions : 
	-  ISR entry SR1 is read, checks `ADDR` flag is high. 
	-  `ADDR` is cleared by reading SR1 followed by SR2 - mandated in RM. 
	-  Write `reg_addr` to DR, sending the target register address on the slave (BME280)
	-  Advance state to REG_SENT
-  Hardware actions : 
	-  Shifts the register address byte out onto the bus. 
	-  Clock stretches SCL low after the byte while waiting for the next DR write or `BTF` condition. 
	-  When the byte has moved from DR into the shift register, `TXE` set high
	-  When both DR and shift register are empty, byte fully transmitted plus ACKed, `BTF` is set high. Asserts event IRQ. 
### STATE : REG_SENT
-  This is where the different operations (r/w) split. 
-  Write path : 
	-  Software actions : 
		-  Confirm `TXE` is high. 
		-  Write `tx_val` to DR. 
		-  Advance to next state in write path - `TX_VAL`. 
	-  Hardware actions : 
		-  `TXE` cleared when DR is written to. 
		-  Peripheral queues `tx_val` behind `reg` in the shift register pipeline. 
		-  When `tx_val` moves from DR to shift register, `TXE` sets again -> next ISR entry. 
-  Read path : 
-  For the read path, we do not want to write another byte to DR. Must wait until `reg` has been fully Tx off the bus, not just moved out the shift register. SO MUST wait for both `TXE` and `BTF`. `BTF` sets high when both DR and shift register are empty and the slave (BME280) has `ACKed` it.
-  On the read path, the ISR will fire when `TXE` is set when `reg` moves to the shift register. Must check op type to ensure it is read, if so do nothing till `BTF` set. When set we must generate a repeated start (keeps bus owned throughout).
	-  Software actions : 
		-  Check `TXE` and `BTF` together. 
		-  Generate repeated start. 
		-  Advance state - `START_R`
	-  Hardware action : 
		-  SCL is clock stretched, held low, while `BTF` is set - bus is paused. 
		-  `START` asserted while stretch is held, hardware generates the repeated `START` condition without releasing bus ownership. 
		-  `SB` sets and event IRQ fires. 
### STATE : TX_VAL
-  Entry from `REG_SENT` write branch. 
-  `TXE` high when `reg` has been moved from the shift register on to the bus and has been ACKed by the slave. `tx_val` has been moved from DR to the shift register, empty DR means `TXE` asserted high by hardware, `TXE` also asserts event IRQ.
-  Do not want to write anything more to DR, the transfer is one address byte (`addr`), one register byte (`reg`) and one data byte (`tx_val`). Need to wait for shift register to be empty so must wait for `BTF`. 
-  `BTF` and `TXE` assert high when `tx_val` is fully transmitted and `ACK`ed. Hardware will stretch SCL and assert the event IRQ. 
-  Software actions : 
	-  Check `BTF` and `TXE` set high. 
	-  Assert `STOP`. 
	-  Restore `ACK`
	-  Advance state - `DONE`
-  Hardware actions : 
	-  `STOP` asserted while SCL stretched - hardware generates `STOP` condition as soon as stretch is released. 
	-  SDA goes high while SCL is high, releasing bus. 
	-  SCL and SDA float back high via the pull-ups.
	-  `SR2.BUSY` clears.
### STATE : START_R
-  Entry from `REG_SENT` read branch. 
-  Repeated start condition has been generated on the bus. SDA pulled low while SCL high. Hardware set `SB` and asserts event IRQ.  Similar to phase 2, however, this time rather than a 7 bit address and a write bit, this phase will send the same address but this time use a read bit. 
-  Software actions : 
	-  Confirm `SB` is set high. 
	-  Write `(addr7 << 1) | 1` to DR - BME280 address with r/w bit set high, switching bus direction to receive.
	-  Advance state - `ADDR_R_SENT`
-  Hardware actions : 
	-  Writing to DR clear `SB` and begins shifting the address byte out onto SDA, MSB first and eight clock cycles. 
	-  On the 9th clock, BME280 must pull SDA low to `ACK` - if it does `ADDR` is set and event IRQ fires. 
	-  BME280 internally notes the r/w bit is high - it prepares to start driving SDA with data on the next clock after `ADDR` is cleared.
### STATE : ADDR_R_SENT
-  `ADDR` set high, slave `ACK`ed the read address. SCL is stretched by hardware, bus is paused. Critical timing - everything that must be configured before the first data bit arrives must happen before `SR2` dummy read that clears `ADDR` and releases the stretch. Moment `ADDR` is cleared and SCL stretch is released BME280 will begin clocking data onto SDA immediately. 
-  Actions will depend on the number of bytes that are being received, `rx_len`. 
- `rx_len == 1` : 
	-  Software actions :
		-  Clear `ACK`, the single incoming byte needs to be NACKed to signal end of transfer. 
		-  Read SR1 then SR2 to clear `ADDR` - SCL releases and byte starts arriving. 
		-  Queue `STOP`, must be queued but will not execute until the byte is fully received.
		-  Advance state - `RX_1`
	-  Hardware actions : 
		-  SCL releases, BME280 clocks one byte onto SDA. 
		-  Peripheral NACKs on 9th clock - BME280 releases SDA. 
		-  `RXNE` set high when byte lands in DR and fires event IRQ. 
-  `rx_len == 2` : 
-  By the time `RXNE` fires for the first byte, the second byte is already clocked into the shift register. No opportunity to clear `ACK` between the two bytes using `RXNE` alone, must also use `POS`. 
-  Normally `ACK` controls whether the byte currently being received is `ACK`ed. `POS` shifts that control forward by one position - `ACK` then applies to the next byte to be received rather than the current one. `POS` must be set before `ADDR` is cleared, it means - `ACK` the first byte and `NACK` the second byte. 
	-  Software actions : 
		-  Set `POS`
		-  Clear `ACK`
		-  Clear `ADDR` - dummy read SR1 then SR2. 
		-  Advance state - `RX_2`. 
		-  No `STOP` queued here must be done in `RX_2`. 
	-  Hardware actions : 
		-  SCL releases, BME280 clocks two bytes. 
		-  First byte ACKed second is NACKed. 
		-  `BTF` sets when both bytes are sitting in DR and shift register respectively, this fires event IRQ. 
-  `rx_len >= 3` : 
-  `ACK` remains set throughout so the slave keeps sending bytes. `STOP` and `ACK` managed as final bytes approach. 
	-  Software actions : 
		-  `ACK` already set from producer, can assert again here to ensure ACK is set. 
		-  Clear `ADDR`, SR1 then SR2 dummy read. 
		-  Advance state - `RX_BULK`. 
	-  Hardware actions : 
		-  SCL releases, BME280 begins streaming bytes. 
		-  Each bytes ACKed, `RXNE` fires per byte, `BTF` fires when DR and shift register both full. 
### STATE : RX_1
-  `RXNE` set high when the single byte is fully received into DR. Hardware asserts event IRQ. `ACK` already cleared before `ADDR` was released so the peripheral `NACK`ed the byte on the 9th clock - the BME280 has already released SDA. The queued `STOP` will soon execute as soon as DR is read. 
-  Software actions : 
	-  Confirm `RXNE` is set high. 
	-  Read DR, hardware clears `RXNE` when DR is read and `STOP` executes. 
	-  Restore `ACK` ready for next transfer. 
	-  Advance state - `DONE`
-  Hardware actions : 
	-  Reading DR clears `RXNE`. 
	-  The queued `STOP` condition executes - SCL goes high, then SDA goes high while SCL is high. 
	-  Bus is released, SCL and SDA float high via external pull ups. 
	-  `SR2.BUSY` clears. 
### STATE : RX_2
-  Check `BTF` is set high here not `RXNE` as want both DR and shift register to be full. Hardware stretches SCL and asserts the event IRQ. 
-  Do not act on `RXNE` here as by the time `RXNE` fires for the first byte the second byte is already being clocked in and it is too late to safely queue `STOP`. 
-  Software actions : 
	-  Check `BTF` is set high. 
	-  Queue `STOP` first, must happen before reading DR else third byte may be clocked. 
	-  Read DR, first byte stored in `rx_buf`,  second byte moved from shift register into DR and `BTF` is cleared. 
	-  Read DR for second byte. 
	-  Clear `POS` - MUST be cleared else risk corrupting next transfer. 
	-  Restore `ACK`
	-  Advance state - `DONE`
-  Hardware actions : 
	-  `STOP` queued - peripheral will generate `STOP` after shift register drains. 
	-  First DR read clears `BTF`, second byte shifted from shift register to DR. 
	-  Second DR read empties DR. 
	-  `STOP` condition executes - SDA goes high while SCL high. 
	-  Bus released, SCL and SDA float high via external pull-ups. 
### STATE : RX_BULK
-  Two distinct behaviours dependent on how many bytes remain. When more than 3 bytes and when only 3 bytes left. 
-  When `rem > 3` : 
	-  Software actions : 
		-  Confirm `RXNE` is set. 
		-  Read DR into handle buffer. 
		-  No change of state required, `rx_idx` will naturally decrement `rem`.
	-  Hardware actions : 
		-  Reading DR clears `RXNE`. 
		-  BME280 ACKed, the next byte begins clocking. 
		-  `RXNE` sets again when the byte lands in DR. 
-  When `rem == 3` : 
	-  Software actions : 
		-  Confirm `BTF`, DR and shift register full and BME280 ACKed, stretching SCL. 
		-  Clear `ACK`, last byte will be `NACK`ed when it arrives. 
		-  Read DR once, releasing SCL so last byte will start being clocked onto SDA. 
		-  Advance state - `RX_2`. 
	-  Hardware actions : 
		-  Reading DR clears `BTF`, releases SCL stretch. 
		-  Second to last byte moved from shift register to DR. 
		-  Last byte starts being clocked in, will be NACKed on the 9th clock. 
		-  `BTF` fires again when shift register filled. 
-  Need two different flag strategies as reading two bytes per `BTF` the whole way breaks for even `n`. So use `RXNE` to read one byte per interrupt till the last three bytes, then there is critical timing on flags. 
-  Why we can use RX_2 state : 
	-  Entry from `RX_BULK` when only 3 bytes remained to be read. One DR read performed in `RX_BULK` meaning that now last two bytes are in DR and shift register and last byte was `NACK`ed. 
	-  `BTF` is set, SCL is stretched. Both final bytes are captured and the bus is paused. Identical mechanics to `RX_2`. 
### Consumer Function - Error Handler
-  Entered upon any error condition flagged in SR1. Four error flags are handled : 
	-  `BERR` - Bus error, illegal `STOP` or `START` detected on bus. 
	-  `ARLO` - Arbitration loss, another master claimed the bus. 
	-  `AF` - Acknowledge failure, slave did not `ACK` an address or data byte. 
	-  `OVR` - Overrun, a new byte arrived before DR is read. 
-  Software actions : 
	-  Read SR1 to identify which flag fired. 
	-  Clear the error flag. 
	-  Assert `STOP` to release the bus. 
	-  Clear `POS` and restore `ACK` to a known state

# Design Notes 

-  **ADDR clear sequence** - reference manual dictates that `ADDR` is cleared by reading SR1 followed by SR2.
-  **Clock stretching** -  when the I2C peripheral needs more time before it is ready to proceed, it holds SCL low pausing the bus. No data changes on SDA while SCL is low, so slave simply waits for release. Allows a slower CPU to communicate with a faster peripheral without data loss.   
-  **ACK and NACK** - after each received byte, the master signals to the slave whether it should send another byte. Pulling SDA low on the 9th clock is an `ACK` - means continue sending. Releasing SDA high on the 9th clock is a `NACK` - means stop sending. 
-  **Atomic transaction** - an atomic transaction is any operation that either fully completes or does not happen at all, there are no observable intermediate states. The key property is that from an outside observer, the operation appears to happen instantaneously. Example : the repeated `START` keeps the register-address write and the data read as one uninterruptible bus transaction, so that no other master can claim the bus and corrupt the slaves internal register pointer between two states. So the outside perspective ie other masters only see before or after state nothing in between.  
-  **Multiple devices on one bus** - every device on the I2C bus sees every `START` condition and every byte that follows it. Each slave compares the transmitted address to its own and ignores the transaction if it does not match. Constraint is that two devices cannot share the same address.   
-  **Pull up and pull down resistors** - Pull up and pull down resistors give a signal line a default state when nothing is actively driving it. Pull up resistors connect the signal line to the suppl voltage through a resistor, default state is high and devices communicate by pulling low (Both SDA and SCL are pulled high using a pull up resistor). Pull downs are less common but useful when the default state is idle condition is logically low.
